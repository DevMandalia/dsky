Architecture

Pre-registered, append-only quant research system. The event log is the source of truth; the projections are derived state; the discipline scans enforce the invariants on every commit.

01

System overview

Five subsystems stack on top of each other. Tests enforce properties; projections read from events; the engine applies the schema; the clock is the only legal source of now.

Figure 1.1 — Layered architecture. Arrows on the right show the direction of dependency and data flow.

Reading direction: the codebase is built bottom-up (clock → schema → engine → events → projections → tests). Code in any layer may import from any layer below it; never the other way.

02

Event log

The single write path. One function, append_event(), is allowed to insert into the events table. Each row's hash is SHA-256(prev_hash · canonical_payload · ts), so any mutation is detectable.

Figure 2.1 — The append_event() call sequence. Dashed return arrows; solid forward arrows.

Why a hash chain? because the only thing that's append-only in SQLite is convention. The hash turns the convention into a fact: a row whose payload was edited post-hoc is no longer consistent with its own hash, and the very next row's prev_hash is no longer consistent either. verify_chain() walks the whole log in one pass and returns the id of the first bad row, or None if the log is intact.

03

Projections

rebuild_projections(conn) is the only writer to any projection table. It truncates them all, then replays the event log in id order, dispatching each event to a handler looked up in _HANDLERS.

Figure 3.1 — rebuild_projections() flow. The implicit transaction guarantees all-or-nothing: either every projection is rebuilt consistently, or the connection is left exactly as it was.

Properties verified by tests:

  • Idempotent — rebuild(rebuild(x)) == rebuild(x)
  • Deterministic — two rebuilds on the same log are byte-identical at the column level
  • Atomic — a mid-rebuild failure rolls back to the prior projection state, not a half-built one
  • Log wins — a corrupted projection row is overwritten on the next rebuild (the log is the source of truth)
04

Clock

The Clock protocol is the only legal source of now. The discipline test scans the entire repo (excluding caches) for any datetime.now() call not on a Clock receiver — and fails the suite if it finds one.

Figure 4.1 — The Clock protocol and its two implementations. Dashed arrows indicate protocol satisfaction (structural subtyping, not inheritance).

Discipline: no module other than src/dsky/clock.py may call datetime.now(), datetime.utcnow(), time.time(), or any equivalent. The whole-repo AST scan in test_clock_discipline.py enforces this; the test goes red the moment anyone slips a secret watch past the gate.

Discipline scans

Two AST-based scans live in the test suite and run on every commit. They turn architectural rules into mechanical checks — substring grep wouldn't catch from datetime import datetime as dt; dt.now(), so we parse the source tree and walk the AST.

06

Time rules

Every event row carries two timestamps. The backtest gate is mechanical: a fact is visible at decision_time iff available_time ≤ decision_time. event_time is metadata; available_time is the gate.

Figure 6.1 — Backtest visibility window. Green dots are readable at decision_time; red dots arrived (or will arrive) too late.

Why two timestamps, not one? in real markets a price print at 14:00:00 is not the same fact as "we received the 14:00:00 price at 14:00:00.873". The first is an event_time, the second is an available_time. A backtest that pretends to be at 14:00:00 must not see the print until 14:00:00.873 — otherwise it has a perfect view of a clock that doesn't exist in production.

Write authority per table:

  • events — db/events.py:append_event() only (test code exempt, by design)
  • current_ideas, rejection_ledger, playbooks — db/projections.py:rebuild_projections() only
  • paper_positions, attribution_history — nothing yet

The write-discipline scan enforces this on src/dsky/ at every commit. A file that gains a raw INSERT outside the allow-list fails the suite.

The rule, restated for the record: when a test is asked for first, the test is written before the implementation, and it fails before the implementation exists. A test is never weakened to make a failing implementation pass. The first response to a green test that shouldn't have been green is to ask, not to celebrate.
